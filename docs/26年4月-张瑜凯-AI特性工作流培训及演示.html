Feature 工作流

AI 辅助开发工作流培训
持久化任务记忆 + 结构化工作流 + 知识沉淀

当前开发流程存在的问题

❌ 无工程级 spec 规约文档

缺少统一的规范文档，且无更新 spec 规约流程

❌ 特性流程无记忆功能

中断后无法恢复，需要重新描述上下文

❌ 需求澄清不够精准

需要多轮头脑风暴才能明确需求

❌ 测试用例未更新需求

导致方案遗漏，需求与测试不一致

❌ AI 问题反复

没有经验沉淀，无知识复用机制

❌ 无文档归档流程

缺少完整的任务闭环

设计原则

Knowledge Compound 解决方案文档化，形成可复用知识库

/start-task
初始化会话，读取项目上下文，判断任务类型，路由到后续 Skill

📋 处理流程

  • 读取 workflow.md 获取工作流上下文
  • 获取当前 developer 信息
  • 获取 Git 状态（status/log）
  • 检查活动任务列表
  • 判断任务类型：简单问题/简单任务/复杂任务
  • 路由到对应后续 Skill

❓ 常见问题

Q: 如何判断任务类型？
A: 根据需求复杂度、是否涉及架构变更、是否需要多轮迭代来判断
Q: 任务中断后如何恢复？
A: 再次运行 /start-task，会读取现有任务状态
Q: 影响哪些文件？
A: 读取 workflow.md、.developer；创建 task.json（新任务）

💡 示例场景

场景1：简单问题
"解释这个函数的作用" → 直接回答
场景2：简单任务
"添加日志工具函数" → 路由到 /init-plan
场景3：复杂任务
"实现用户认证系统" → 路由到 /brainstorm
/brainstorm
对复杂任务进行需求澄清，生成 PRD 文档，对比方案，识别风险

📋 处理流程

  • 创建或选择任务目录
  • 使用 GitNexus MCP 查询项目上下文
  • 查询代码结构和模式
  • 对架构选型给出 2-3 个方案
  • 收敛 MVP 范围
  • 定义验收标准
  • 识别风险点
  • 生成 prd.md
  • 等待人工校验 PRD

❓ 常见问题

Q: PRD 不准确怎么办？
A: 提供反馈后重新运行 /brainstorm
Q: 需要多少轮迭代？
A: 通常 1-2 轮即可收敛
Q: 输出什么文件？
A: prd.md

💡 示例场景

场景：用户认证系统
方案对比：JWT vs Session
MVP范围：登录、注册、权限检查
验收标准：功能完整、安全、性能达标
风险：密码存储、Token安全
/init-plan
作为轻量任务的工作流入口，创建或恢复任务目录、最小 PRD 和三文件工作记忆

📋 处理流程

  • 检查 PRD 是否存在
  • 检查三文件是否存在
  • 创建 task_plan.md（任务计划）
  • 创建 findings.md（研究发现）
  • 创建 progress.md（进度记录）
  • 更新 task.json 状态为 planning

❓ 常见问题

Q: 三文件是什么？
A: task_plan.md（任务计划）、findings.md（研究发现）、progress.md（进度记录）
Q: 什么时候调用？
A: 简单任务在 /start-task 后，复杂任务在 /brainstorm 后
Q: 输出什么文件？
A: prd.md task_plan.md findings.md progress.md

💡 示例场景

场景：添加日志工具函数
简单任务流程：
/start-task → /init-plan → 直接实现
/research
在编写实施计划前进行代码库调研，建立代码事实基线

📋 处理流程

  • 检查 GitNexus 索引状态
  • 索引不可用则执行 analyze
  • 查询相似模块
  • 查看关键符号调用者
  • 查看被调用者关系
  • 查看所属流程
  • 预估改动范围 impact
  • 将结论写入 findings.md
  • 回写 prd.md 中的约束
  • 回写 task_plan.md 中的风险

❓ 常见问题

Q: GitNexus 索引是什么？
A: 代码知识图谱，包含调用关系、类继承、模块结构等
Q: 如何建立索引？
A: 运行 npx gitnexus analyze
Q: 输出什么文件？
A: 更新 findings.md

💡 示例场景

场景：用户认证系统调研
发现：现有用户模型结构
发现：权限系统已实现部分功能
发现：JWT 库版本过旧需升级
风险：数据库迁移影响范围大
/before-dev
读取相关规范，统一可执行上下文，确保开发准备就绪

📋 处理流程

  • 检查三文件是否初始化
  • 未初始化则调用 /init-plan
  • 检查是否已做代码研究
  • 未研究则调用 /research
  • 读取 spec/<package>/<layer>/index.md
  • 读取 spec/guides/index.md
  • 读取前置开发检查清单
  • 明确本次涉及的模块
  • 明确必须遵守的 spec
  • 明确必须验证的风险点

❓ 常见问题

Q: spec 文件在哪？
A: .feature/spec/<package>/<layer>/index.md
Q: 简单任务需要吗？
A: 可选，复杂任务必须执行
Q: 输出什么？
A: 开发上下文就绪状态，更新 task.json

💡 示例场景

/write-plan
生成可执行的实施计划，按任务拆分到文件级别

📋 处理流程

  • 读取 prd.md
  • 读取 findings.md
  • 读取相关 spec
  • 生成结构化实施计划
  • 为每个任务指定文件路径
  • 为每个任务定义测试动作
  • 为每个任务定义验证命令
  • 应用 TDD/YAGNI/DRY 原则
  • 生成 implementation-plan.md
  • 等待人工校验 Plan

❓ 常见问题

Q: Plan 不准确怎么办？

Q: 颗粒度如何？
A: 拆分到文件级别，适合代理执行
Q: 输出什么文件？
A: implementation-plan.md

/write-testcase
基于任务文档生成标准化的功能测试用例，复杂任务必须执行

📋 处理流程

  • 读取任务文档（prd、findings、plan）
  • 读取测试规范和模板
  • 识别产品类型（PK/HY/XF）
  • 生成需求分析报告
  • 设计测试用例
  • 生成 testcase_analysis.md
  • 生成 testcase.md
  • 更新 task.json

❓ 常见问题

Q: 如何识别产品类型？
A: 根据关键词自动识别，也可手动选择
Q: 产品类型有哪些？
A: PK（屏控）、HY（会议）、XF（信发）
Q: 输出什么文件？
A: testcase_analysis.md testcase.md

💡 示例场景

产品类型识别

HY: 会议平板/OPS/摄像头
XF: 信息发布平台/终端软件
/check-testcase
基于产品类型验证测试用例完整性，补充遗漏测试点，更新任务文档

📋 处理流程

  • 读取任务和测试文档
  • 确认产品类型
  • 读取产品规格和 Checklist
  • 筛选相关模块
  • 执行模块验证
  • 补充遗漏用例
  • 生成 testcase_checkdetail.md
  • 评估是否涉及新需求
  • 更新所有任务文档

❓ 常见问题

Q: 发现新需求怎么办？
A: 同步更新 prd.md、task_plan.md、implementation-plan.md
Q: 覆盖率要求？
A: 建议覆盖率 >80%
Q: 输出什么文件？
A: testcase_checkdetail.md + 更新所有任务文档

💡 示例场景

场景：发现遗漏测试点

遗漏：并发登录限制测试

/executing-plans
在当前会话内按计划顺序推进，适合耦合较强的任务

📋 处理流程

  • 读取 implementation-plan.md
  • 解析任务列表
  • 按顺序执行任务
  • 涉及重大修改时调用 /impact
  • 更新 task_plan.md
  • 更新 progress.md
  • 关键发现更新 findings.md
  • 完成所有任务后准备检查阶段

❓ 常见问题

Q: 和 /subagent-work 的区别？
A: executing-plans 在当前会话内执行；subagent-work 启动独立子代理
Q: 什么时候用？
A: 任务耦合性强、需要实时交互时使用
Q: 输出什么？
A: 代码修改 + 更新 progress.md

💡 示例场景

场景：顺序实现认证系统
Task1 → Task2 → Task3 → ...
每个任务完成后更新 progress.md
发现问题时更新 findings.md
/subagent-work
使用独立子代理执行任务，适合独立性强、需长时间自治开发的任务

📋 处理流程

  • 读取 implementation-plan.md
  • 解析任务列表
  • 启动子代理执行任务
  • Spec reviewer 检查 PRD 一致性
  • Code quality reviewer 检查代码质量
  • 不合格则返回修复
  • 合格则进入下一任务
  • 更新 task_plan.md
  • 更新 progress.md

❓ 常见问题

Q: 什么时候用？
A: 任务独立性强、可以并行执行时使用
Q: 子代理是什么？
A: 独立的 AI 会话，可以长时间自治开发
Q: 评审流程？
A: Spec reviewer → Code quality reviewer

💡 示例场景

场景：并行开发认证系统
子代理1：数据库迁移
子代理2：认证API
子代理3：前端登录页

/impact
在修改前评估影响范围，识别风险等级

📋 处理流程

  • 识别目标符号
  • 执行 GitNexus impact 分析
  • 查看 upstream impact（直接调用者）
  • 查看相关 processes（执行流）
  • 计算 blast radius（影响范围）
  • 评估风险等级
  • 生成影响报告

❓ 常见问题

Q: 风险等级有哪些？
A: LOW（允许修改）、MEDIUM（需评审）、HIGH（需架构师批准）、CRITICAL（禁止修改）
Q: 什么时候运行？
A: 修改公共函数、核心模块、基础组件前
Q: 输出什么？
A: 影响范围报告 + 更新 task.json

💡 示例场景

场景：修改用户模型
影响：15 个文件
调用者：AuthService, UserService, OrderService
风险等级：MEDIUM
建议：需要评审后修改
/check
进行轻量到中度的质量检查，面向开发中场景

📋 处理流程

  • 获取代码变更
  • 自动发现相关 spec 模块
  • 执行 lint 检查
  • 执行 type 检查
  • 执行单元测试
  • 检查规范对齐
  • 检查代码风格
  • 输出检查结果

❓ 常见问题

Q: 和 /review 的区别？
A: /check 是轻量自检，/review 是正式多维度评审
Q: 检查失败怎么办？
A: 列出错误详情，修复后重新运行 /check
Q: 输出什么？
A: 检查结果 + 更新 task.json（checkPassed）

💡 示例场景

场景：认证系统自检
✓ Lint 检查通过
✓ Type 检查通过
✓ 单元测试通过（覆盖率 85%）
✓ 规范对齐通过
💡 最佳实践 Tips
开发自检后建议切换模型进行正式评审：
• 自检可由开发模型完成，确保基础质量
• 正式评审建议使用不同的模型，获得更客观的视角
• 推荐组合：GLM-5.0 开发 + Kimi-K2.5 评审
/review
进行多角色正式评审，包括正确性、可维护性、安全性等多个维度

📋 处理流程

  • 收集评审材料
  • 代码变更
  • PRD 文档
  • spec 规范
  • 启动多角色评审
  • correctness 检查
  • maintainability 检查
  • testing 检查
  • security 检查
  • performance 检查
  • 汇总评审结果

❓ 常见问题

Q: 评审维度有哪些？
A: 正确性、可维护性、测试覆盖、安全性、性能、规范对齐、影响范围、关联流程、文档完整
Q: 评审不通过怎么办？
A: 返回执行阶段修复问题
Q: 输出什么？
A: 评审报告 + 更新 task.json（reviewPassed）

💡 示例场景

场景：认证系统评审
✓ 正确性：逻辑正确
✓ 安全性：密码哈希正确、Token 安全
✓ 测试覆盖：85%
⚠ 性能：登录接口需优化
→ 返回优化性能
💡 最佳实践 Tips
推荐使用不同模型进行开发与评审：
• 开发阶段：使用 GLM-5.0 等模型进行代码编写
• 评审阶段：使用 Kimi-K2.5 等不同模型进行独立评审
• 模型越强大，评审质量越高，交叉验证效果越好
• 也可以 CodeBuddy + Lingma 配合使用，同时打开同一工程，一个负责开发，一个负责评审
/finish-work
汇总完成条件，校验所有交付物，准备提交

📋 处理流程

  • 执行 detect_changes 分析变更
  • 检查变更影响是否符合预期
  • 判断是否需要更新 spec
  • 检查交付文档完整性
  • 准备进入 compound 阶段

A: 代码、PRD、实施计划、测试用例、文档等
Q: 什么时候更新 spec？
A: 当解决方案揭示可复用规则时
Q: 不完整怎么办？
A: 返回补充交付文档

💡 示例场景

场景：认证系统收尾

/compound
复盘本轮工作，将解决方案文档化，沉淀到项目知识库

📋 处理流程

  • 重读任务产物和代码变更
  • 识别最佳学习要点
  • 分类问题类型
  • 选择追踪轨道（Bug/Knowledge）
  • 读取 schema.yaml 验证字段
  • 搜索已有解决方案
  • 创建或更新解决方案文档
  • 判断是否揭示可复用规则
  • 可选更新 .feature/spec/

❓ 常见问题

Q: Bug Track 和 Knowledge Track？
A: Bug Track 用于 bug/错误/性能问题；Knowledge Track 用于最佳实践/工作流/指导
Q: 输出到哪里？
A: .feature/solutions/<category>/<filename>.md
Q: 如何避免重复？
A: 先搜索已有解决方案，有重叠则更新

💡 示例场景

场景：沉淀认证系统经验

# 进入工程目录 cd /path/to/your-project # 初始化 Feature（CodeBuddy 平台） feature init --codebuddy -u your-name # 其他平台选项： # feature init -u your-name # Claude Code（默认） # feature init --iflow -u your-name # iFlow # feature init --cursor -u your-name # Cursor # feature init --opencode -u your-name # OpenCode

支持的 AI 平台

平台 命令参数 特性支持
Claude Code 默认 完整支持（Hooks）
CodeBuddy --codebuddy 完整支持（Hooks）
iFlow --iflow 完整支持（Hooks）
Cursor/OpenCode --cursor/--opencode 部分支持

目录结构

.feature/ ├── spec/ # 项目规范和标准 │ ├── frontend/ # 前端规范 │ ├── backend/ # 后端规范 │ ├── guides/ # 思维指南 │ └── unit-test/ # 测试规范 ├── tasks/ # 任务跟踪和文档 │ ├── <MM-DD-task-name>/ # 活动任务 │ │ ├── task.json # 任务状态和元数据 │ │ ├── prd.md # 产品需求文档 │ │ ├── implementation-plan.md # 实施计划 │ │ ├── task_plan.md # 任务计划 │ │ ├── findings.md # 研究发现 │ │ ├── progress.md # 进度记录 │ │ ├── testcase.md # 测试用例 │ │ └── testcase_checkdetail.md # 测试检查报告 │ └── archive/ # 已归档任务 ├── solutions/ # 解决方案文档库 ├── workspace/ # 开发者工作区 ├── scripts/ # 工作流脚本 ├── workflow.md # 工作流文档 └── config.yaml # 配置文件
